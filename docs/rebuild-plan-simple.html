What we're doing to rhillane.com, and why

Today, every page of the site is built by hand inside Elementor, a drag-and-drop page builder. We're replacing that with a system of ready-made building blocks that live in our own theme code. Editors fill in content; the code draws the page. Same look, same Google rankings — but faster pages, and a site we fully control.

First deadline Wednesday 22 July, 17:00 The building-block system + the first Case Studies, live on the staging site.
Five terms, in plain words
Elementor
The drag-and-drop page builder the site uses today. The problem: every page's layout is trapped inside it — slow, fragile, hard to maintain.
CPT — Custom Post Type
A new content type in the WordPress admin. Just like "Posts" exists for the blog, we add "Case Studies" and later "Services" as their own menus.
ACF Flexible Content
The building blocks. An editor opens a case study and stacks sections — hero, text, gallery, stats, quote — and fills them in. No design work, no page builder. Our theme code turns that stack into the finished page.
Staging
A private copy of the site (on the staging subdomain) where we build and test everything. The public site is untouched until we deliberately switch.
noindex
A per-page switch that tells Google "don't list this page yet". We use it so old and new versions never compete in search.
Already decided (20/07 meeting)
  • Rank Math is our SEO plugin — it holds each page's Google title, description and settings.
  • The theme is switched on for the whole multisite network at once; content stays separate per site.
  • rhillane.com is the main site — the target for all of this work.
  • Six services, confirmed (Part III).
  • Header and footer are the same across the whole site; a content type can get its own only if it truly needs one.

1.2How the system works

Think of it as three simple parts:

  1. The blocks. A library of about ten section types — hero, text, text-with-image, gallery, stats, quote, logos, call-to-action, features, FAQ. Each block knows what it needs (a heading, an image, a button…) and each has a small spacing/background control, replacing what Elementor used to do.
  2. The editor experience. An editor opens a case study, stacks the blocks they want in order, fills them in, hits publish. No design decisions, no page builder — just content in labelled fields.
  3. The drawing code. One small function in the theme reads the stack and renders each block with the site's real design (colours and fonts copied from the current site, so pages look the same as today). Every page template is literally: header → draw the blocks → footer.

Why this is powerful: adding a brand-new block type to the whole site = one block definition + one small template file. And because block definitions are saved as files in the repository, every change is tracked and deploys automatically when we push to GitHub.

Header & footer: one header and one footer serve the entire site, drawn on every page automatically. If a content type ever genuinely needs its own (say, a special case-study header), WordPress lets us add one just for it — and everything else keeps using the shared pair.

PART II

The method — how every page moves safely

same 3 steps, every time

After this week, we start moving existing Elementor pages onto the new system. Every single page follows the same three steps — this is how we guarantee Google never notices the switch.

  1. Take a blueprint of the old page. We export the Elementor page as a file that records its exact layout — section order, spacing, colours, text. We never load this file back into WordPress; it's the reference we rebuild from, and the "does it look the same?" checklist for review.
  2. Rebuild it with blocks — at the same web address. The page keeps its exact URL whenever possible (for Google, same address + same content = nothing happened). Its Google title, description and structured data are copied over into Rank Math. Only if an address truly must change do we add a permanent redirect from old to new.
  3. Check, then switch. We scan the old page and the new one and compare them line by line — titles, headings, text, links, speed. Only when everything matches do we make the switch on the live site, then re-scan and watch Google's data for a while.

The safety trick — "hidden until ready". When a rebuilt page goes onto the live site, it starts hidden from Google (noindex, and left out of the sitemap) while the old Elementor page keeps its place in search. We test against the hidden version. At switch time — page by page: retire the old Elementor page → point its address at the new one → un-hide the new page → update the sitemap. Old and new are never visible to Google at the same time, and until the switch, going back costs nothing.

2.1The checklist every page must pass before switching

Same web address (or a clean redirect)
Same Google title & description
Same main heading and heading order
No paragraph of text lost
Social-sharing & technical tags unchanged
Structured data still present and valid
Image descriptions (alt text) kept
All internal links still work
Page speed as good or better
Sitemap intact, no broken links created

A group of pages goes live only when every page in it passes every line. Anything less waits.

PART III

The waves — what moves, in what order

after Wednesday

We move the site in five waves, easiest first — so by the time we touch the most important pages, the system has already proven itself several times. Each wave can ship on its own, and each can be undone on its own.

W0The system + Case Studiesthis week — Wed 22/07

Everything in Part I. New content, nothing to protect — the proof that the machine works.

Done when: a non-developer can build a case study from blocks on staging.
W1The blogfirst real migration

Blog posts all share one shape, and no single post is business-critical — the perfect first migration. We build the blog listing and article templates on the block system, move every post across (keeping address, title, date, author, categories), run the checklist, switch, and retire Elementor for the blog.

Done when: every post runs on the new system and passes the checklist.
W2The six services

Services become their own content type, built from the same blocks. One page per service:

Website creationSEOSocial mediaGraphic designAdsMedia buying

All six are built hidden from Google first, at addresses matching today's service pages. Then, service by service: once a rebuilt page passes the checklist, its old Elementor page is retired and the new one takes its place — in small waves, never all at once.

Done when: all six pass the checklist and no Elementor service page is left.
W3Homepage + shared parts

The homepage, the header and the footer — the highest-traffic, highest-risk surfaces — move last, once the system has proven itself three times over.

Done when: a full-site scan shows no differences that matter.
W4Say goodbye to Elementor

With nothing left depending on it, Elementor is removed from the migrated content entirely, followed by a final speed pass. This is the payoff: a lighter, faster site.

Done when: Elementor no longer draws a single migrated page.
PART IV

House rules

  • Naming is consistent — every block's name matches its template file exactly, so the system finds things by convention, not configuration.
  • Blocks are never edited only on the live site — always: build on staging → save to the repository → deploy. That way every change is reviewed and reversible.
  • One branch per wave, and the before/after scan is attached to the review as proof.
  • Reference files (Elementor blueprints) never get deployed — they stay out of the live theme folder.
